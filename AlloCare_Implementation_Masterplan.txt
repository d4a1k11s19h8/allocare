1. Project Overview & Problem Statement
1.1  The Real Problem (Verbatim from Official Brief)
Critical Distinction — Most Teams Will Get This Wrong
This is NOT about solving lack of resources. It is about solving MISALLOCATION of resources due to POOR VISIBILITY. Teams that build a generic volunteer listing platform will fail the judge brief. The solution must create visibility, intelligence, and automated routing — not just a bulletin board.
Local NGOs and community groups collect valuable need-data through paper surveys, WhatsApp messages, and field reports. This data sits siloed in physical files, disconnected spreadsheets, or a single coordinator's memory. No system currently aggregates it to surface urgency patterns — and no intelligent bridge exists to route available volunteers to where they are needed most, in real time.
Three Compounding Failures
DATA IS UNDIGITIZED: Paper surveys and handwritten field reports cannot be analyzed at scale. Critical patterns — seasonal hunger spikes, recurring flood areas — are invisible to any digital tool.
DATA IS SILOED: Even when digitized, need-data lives across WhatsApp threads, individual Google Sheets, and email chains. No unified view exists to compare urgency across areas or organizations.
MATCHING IS MANUAL: Volunteer coordinators assign tasks through personal knowledge and phone calls. This is slow, non-scalable, language-limited, and systematically misses the best-fit assignments.
1.2  Solution Approach
AlloCare (Allocation + Care) is a three-layer intelligent pipeline: (1) multi-source data ingestion converting paper, CSV, and digital forms into structured need records; (2) a Gemini-powered urgency intelligence engine that scores, clusters, and visualizes needs geographically; and (3) an algorithm-driven volunteer matching system that deploys the right person to the highest-priority task in real time.

3.2  Business Logic Specifications
Urgency Score Calculation
urgency_score = (severity_score × log(report_frequency + 1)) / max(1, days_since_first_report)
normalized_score = min(100, round(urgency_score × 10))
severity_score: Gemini returns 1-10 (1=minor, 10=life-threatening)
report_frequency: count of same issue_type × zone in last 30 days
days_since_first_report: staleness penalty reduces urgency over time
Severity thresholds: 0-30 = Low (green), 31-60 = Medium (amber), 61-85 = High (orange), 86-100 = Critical (red). Critical needs trigger push notifications to all eligible nearby volunteers immediately.
Volunteer Match Score Calculation
skill_overlap_score = (matched_skills / required_skills) ∈ [0, 1]
proximity_score = 1 / (1 + distance_km) — Google Maps Distance Matrix API
availability_score = 1 if volunteer available at need's time window, else 0.3
match_score = skill_overlap_score × proximity_score × availability_score
Returns top-3 volunteers sorted by match_score descending

6. Algorithms & Technical Implementation
6.1  Urgency Scoring Algorithm
Pseudocode
function calculate_urgency_score(need_record):
  severity = need_record.severity_score  // 1-10 from Gemini
  freq = count_reports(same zone, same issue_type, last_30_days)
  days = (now - need_record.first_reported_at).days
  raw = (severity * log(freq + 1)) / max(1, days)
  normalized = min(100, round(raw * 10))
  if normalized >= 86: label = "CRITICAL", color = "#E02424"
  elif normalized >= 61: label = "HIGH",     color = "#F97316"
  elif normalized >= 31: label = "MEDIUM",   color = "#E3A008"
  else:                  label = "LOW",      color = "#0E9F6E"
  return {score: normalized, label, color}
Complexity: O(k) per record where k = recent reports for that zone/issue pair. Runs in Firebase Cloud Function on every new Firestore write. Total latency: < 800ms including Gemini API call.
6.2  Volunteer Matching Algorithm
Pseudocode
function match_volunteers(need_record, volunteer_pool):
  candidates = []
  for v in volunteer_pool where v.status == "available":
    skill_score = len(intersect(v.skills, need.required_skills))
                  / len(need.required_skills)  // 0 to 1
    dist_km = distance_matrix_api(v.location, need.location)
    proximity_score = 1 / (1 + dist_km)         // decays with distance
    avail_score = 1.0 if v.available_now else 0.3
    if dist_km > v.max_distance_km: continue    // hard cutoff
    match_score = skill_score * proximity_score * avail_score
    candidates.append({volunteer: v, score: match_score,
      explanation: build_explanation(v, need, skill_score, dist_km)})
  return sorted(candidates, key=score, desc=True)[:3]
Batch Distance Matrix API call: group all candidate origins into a single API call to minimize latency. Google Maps Distance Matrix returns driving distances for up to 25 origins × 1 destination per call.

6.4  Trend Detection Algorithm
function detect_trends(zone, issue_type, lookback_days=30):
  week1 = count_reports(zone, issue_type, days 22-30 ago)
  week2 = count_reports(zone, issue_type, days 15-21 ago)
  week3 = count_reports(zone, issue_type, days 8-14 ago)
  week4 = count_reports(zone, issue_type, days 1-7 ago)
  slope = linear_regression([week1, week2, week3, week4])
  if slope > 0.5: return "RISING", "⬆ Trend: Increasing urgency"
  elif slope < -0.5: return "FALLING", "⬇ Trend: Improving situation"
  else: return "STABLE", "→ Trend: Stable"

9. Backend Architecture
9.1  System Architecture Overview
AlloCare uses a serverless, event-driven architecture on Firebase. No dedicated backend server required — all processing happens in Firebase Cloud Functions triggered by Firestore writes and HTTP endpoints. This eliminates infrastructure management during the hackathon while remaining production-scalable.
Architecture Layers (top to bottom)
Client Layer: Flutter Web (NGO Dashboard) + Flutter Mobile (Volunteer App)
API Gateway Layer: Firebase Cloud Functions HTTP endpoints (REST)
Intelligence Layer: Gemini API + Cloud Vision API + Google Translate API
Data Layer: Firestore (primary) + Firebase Storage (files) + BigQuery (Phase 2 analytics)
Notification Layer: Firebase Cloud Messaging (push) + Twilio (SMS/WhatsApp)

11. Data Pipeline Architecture
11.1  Ingestion Pipeline
Four parallel ingestion paths converge at the same processing pipeline. All paths produce a raw need_report document in Firestore with source field set and raw_text populated. The onReportCreated Cloud Function handles all paths identically from that point forward.
Path A: Photo Upload
User uploads photo → Firebase Storage → onReportCreated triggers
→ Cloud Vision API (OCR) → extracted_text
→ Translate API (if non-English) → english_text
→ Gemini extraction prompt → structured JSON
→ Maps Geocoding API → {lat, lng}
→ Urgency scoring → normalized_score
→ Firestore update → FCM to nearby volunteers
Path B: CSV Import
CSV file upload → HTTP Cloud Function → pandas parsing
→ Column normalization → batch Firestore write
→ Each new doc triggers onReportCreated normally
Path C: Manual Text Entry
Web form submission → HTTP POST → Firestore write with raw_text
→ onReportCreated triggers → same pipeline as Path A
Path D: WhatsApp/SMS
Twilio webhook → HTTP Cloud Function → parse body
→ Lookup sender number → Firestore write
→ onReportCreated triggers → same pipeline

11.3  Data Quality & Validation
OCR confidence check: if Vision API confidence < 0.6 for any field, flag report for manual review rather than auto-processing
Geocoding validation: if Maps API returns confidence < 0.7, ask coordinator to confirm location on map before saving
Duplicate detection: if same zone + same issue_type + similar summary submitted within 48 hours, merge reports and increment frequency counter instead of creating new record
Human-in-loop override: coordinator can edit any extracted field and submit correction; corrections logged to urgency_corrections collection for model improvement data
12. Integration & Connectivity
12.1  Event-Driven Architecture
AlloCare uses Firestore document lifecycle events as the event bus. This eliminates message queue infrastructure while maintaining asynchronous, decoupled processing.
Key Events

need_report.urgency_score updated → triggers: heatmap refresh (client real-time listener), leaderboard recalculation
assignment.status = "completed" → triggers: volunteer impact stats update, need report slot decrement, impact scorecard generation
volunteer.status = "available" → triggers: check for unmatched critical needs in their zone

16. Additional Essential Elements
16.1  Security Considerations
Firestore Security Rules: All reads/writes require authenticated Firebase user. Coordinators can only read/write their org's data. Volunteers can only write to assignments collection (their own). Admin role required for cross-org reads.
API Key Security: All Google API keys stored in Firebase Environment Config, never in client code. Cloud Functions access keys server-side only.
PII Protection: Volunteer lat/lng rounded to 2 decimal places (±1.1km) before storing — prevents precise location tracking. Full location only used in Distance Matrix calls, never persisted.
Data Privacy: Field reports contain community data, not individual PII. Photo uploads stripped of EXIF metadata before storage. GDPR-compliant delete flow on volunteer account deletion.
Rate Limiting: Cloud Functions HTTP endpoints rate-limited at 100 req/min per IP via Firebase App Check.
16.2  Accessibility Requirements
WCAG 2.1 AA: All text meets 4.5:1 contrast ratio minimum. Critical alerts use color + icon (never color alone).
Screen reader support: All Flutter widgets use Semantics wrapper with meaningful labels.
Low-bandwidth mode: Images lazy-loaded. Heatmap tiles cached. CSV ingestion works on 2G connections.
Offline support: Flutter app uses Firestore offline persistence. Reports submitted offline queue and sync on reconnect.
